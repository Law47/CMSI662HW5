# CMSI 662 Homework 5: Secure Webapp Project

This repository contains a small Flask banking application based on the notes in https://cs.lmu.edu/~ray/notes/websecurity/. It demonstrates common web security defenses in an example banking web app.

## What is Included

- Login with PBKDF2-SHA256 password verification through `passlib`
- Salted password hashes generated by `bin/createdb.py`
- JWT authentication tokens stored in `HttpOnly` cookies
- Account details visible only to the logged-in account owner
- Completed transfer flow with source ownership checks, target checks, amount
  validation, balance checks, and an atomic database transaction
- Flask-WTF CSRF protection on every form
- Jinja templates that autoescape user-controlled output to defend against XSS
- Generic login errors and a dummy password hash verification path to reduce user enumeration signals

## Setup

Create and activate a virtual environment, then install dependencies:

```powershell
python -m venv .venv
.\.venv\Scripts\Activate.ps1
pip install -r requirements.txt
```

Create the demo SQLite database:

```powershell
python bin\createdb.py
```

Run the app:

```powershell
flask --app app run
```

Open <http://127.0.0.1:5000>.

| `bob@example.com` | `123456` | `998`, `777` |

These passwords are intentionally weak because this is a demo. The storage mechanism is the important part: the database stores salted PBKDF2 hashes, not plaintext passwords.

## Security Notes

- **XSS:** HTML is rendered through Jinja templates, which escape variables by
  default.
- **CSRF:** Flask-WTF requires a valid CSRF token on state-changing form posts.
- **SQL Injection:** All SQL uses `?` placeholders with bound parameters.
- **Authentication:** A signed, expiring JWT is stored in an `HttpOnly` cookie.
- **Authorization:** Account balance and transfer operations check ownership on
  the server side.
- **User enumeration:** Login returns one generic error and performs dummy
  password-hash work when an email is missing.
- **Validation:** Account numbers and transfer amounts are checked before the
  service layer performs database updates.

For a production app, use HTTPS, strong secrets from environment variables,
strong password rules, multi-factor authentication, monitoring, automated tests,
and a real database deployment with least-privilege credentials.
